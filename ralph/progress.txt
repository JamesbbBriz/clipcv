## Codebase Patterns (will accumulate as work lands)

- **Build pipeline** is Vite 8 + `@crxjs/vite-plugin` 2.4. Manifest is the
  canonical source of build entry points: list any new background /
  content / popup / options entry in `src/manifest.json` and CRXJS picks
  it up automatically. Do NOT add `rollupOptions.input` manually.
- **Manifest path convention**: in `src/manifest.json`, refer to source
  files using their repo-relative path (`src/background/service-worker.ts`)
  for code, and the public-folder-relative path (`icons/16.png`) for
  static assets. CRXJS rewrites both to the correct `dist/...` paths at
  build time.
- **Icons live in `public/icons/`** and are emitted to `dist/icons/`.
  Placeholder 1x1 PNGs are committed for now; real 16/48/128 PNGs land
  in US-013.
- **Service worker entry**: `src/background/service-worker.ts` with
  `type: "module"` in the manifest. The SW is killed between events —
  never hold long-lived in-memory state (US-005+ message handlers must
  be event-driven).
- **TypeScript**: strict + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`.
  Path alias `@/*` → `src/*` is wired in `tsconfig.json` and `vite.config.ts`.
- **Dev server port**: 5179 (5173 collides with other local dev). HMR
  auto-rebuilds the extension; reload the extension in chrome://extensions
  to pick up SW changes.
- **No default exports** in new modules — named exports only (CLAUDE.md §9).
- **Popup entry naming**: HTML-side script ref is `./main.tsx`; the React
  component lives in `Popup.tsx`. Rationale: macOS' case-insensitive FS
  collides `popup.tsx` with `Popup.tsx`, and `forceConsistentCasingInFileNames`
  amplifies this. Future entry points (options page, content script) follow
  the same `main.tsx` + `<ComponentName>.tsx` split.
- **Tailwind**: v4 via `@tailwindcss/vite` plugin (no PostCSS config, no
  tailwind.config.js needed). Each entry point owns a `*.css` file that
  starts with `@import 'tailwindcss';`. CRXJS picks up the CSS through
  the entry HTML's `<link>` (or implicitly via the JS entry's CSS import).
- **Manifest action block**: `action.default_popup` uses the same
  `src/<entry>/<file>.html` repo-relative path; CRXJS rewrites it to
  the bundled `dist/src/<entry>/<file>.html` at build time.
- **Service worker lifecycle**: minimum coverage is
  `onInstalled` (install/update) + `onStartup` (browser launch / "activate")
  + `onSuspend` (just-before-teardown). Listeners must remain stateless —
  no module-scope mutable state.
- **Verification of UI stories**: programmatic structural check of `dist/`
  (manifest fields, popup.html shape, bundle contents, forbidden-pattern
  grep) is the documented AC verifier. Chrome 137+ blocks
  `--load-extension` from CLI without policy/dev-mode, so automated
  unpacked-load is unreliable; the build artifacts + structural checks are
  the source of truth.
- **Path alias under spaces**: the repo path `OptiTalent Nan V1/clipcv`
  contains a space. `new URL('./src', import.meta.url).pathname` returns
  the URL form (`%20`-encoded), which Rolldown can't resolve. Use
  `decodeURIComponent(...)` (or `fileURLToPath` if `@types/node` is
  available — currently it isn't) when wiring resolve.alias in
  `vite.config.ts`.
- **BYOK at-rest encryption** (`src/lib/crypto/byok-crypto.ts`): AES-GCM,
  256-bit raw key generated once per install via
  `crypto.subtle.generateKey({name:'AES-GCM', length:256}, true, ...)` and
  exported as base64; per-call random 12-byte IV. The raw key, ciphertext
  and IV all live in `chrome.storage.local` under the single key
  `byok_settings` (see `src/lib/storage/byok-store.ts`). Plaintext API
  key is never persisted. The module exports `generateRawKey`,
  `encryptString`, `decryptString` only — no shared state, no module-scope
  CryptoKey cache (the SW is killed between events).
- **WebCrypto type variance**: under TS 5.7+ `Uint8Array` defaults to
  `Uint8Array<ArrayBufferLike>`, but lib.dom's `crypto.subtle.encrypt`
  signature requires `ArrayBuffer`-backed views. We allocate a fresh
  `ArrayBuffer` copy via a small `toArrayBuffer(view)` helper at every
  WebCrypto call site — see `byok-crypto.ts`. Future stories that touch
  WebCrypto / fetch body should reuse this pattern.
- **Secret scrubber** (`src/lib/util/scrub-secrets.ts`): single regex
  pass over any string before display: `/sk-[A-Za-z0-9]{16,}/g` and
  `/Bearer\s+[A-Za-z0-9._-]{20,}/g` → `[redacted]`. Apply in every
  user-facing error path (per CLAUDE.md §6.2). The options page test-
  connection error display already routes through it; reuse the same
  helper from US-008 onwards when surfacing LLM errors.
- **Manifest permissions**: stayed minimal. Only `"storage"` requested so
  far (US-003). `<all_urls>` host permission lands with the content
  script in US-004; downloads + activeTab in US-011. Each story adds the
  narrowest required permission only.
- **Options page entry** mirrors the popup convention:
  `src/options/options.html` + `main.tsx` (createRoot bootstrap) +
  `<Component>.tsx` (form). Wired through manifest's `options_page`
  field. Tailwind imported via `options.css`. Future entry points
  (content script in US-004) follow the same `main.tsx` +
  `<Component>.tsx` split.
- **Cross-context messaging contract** (`src/lib/messaging/messages.ts`):
  every chrome.runtime.sendMessage payload is a discriminated union
  (`ClipcvMessage`) and has a matching type guard (`isClipcvMessage`).
  The service worker rejects any unknown `kind` with
  `{kind:'error', reason:'unknown_kind'}`. Receivers MUST extend the
  guard before handling new variants — do NOT add ad-hoc shape checks
  at call sites.
- **Content script structure**: entry `src/content/inject.tsx`
  (idempotent shadow-DOM mount) + component `<FloatingButton />` +
  stylesheet `inject.css`. The CSS is imported with the Vite `?inline`
  query suffix and injected as a `<style>` text node into the shadow
  root, so Tailwind utilities work inside the shadow boundary without
  leaking to the host page (no separate `inject*.css` is emitted by
  CRXJS). `:host { all: initial }` plus `host.style.all = 'initial'`
  block bidirectional CSS bleed.
- **Idempotent shadow-DOM mount**: stable host element id
  (`clipcv-floating-host`) + `getElementById` guard before insert
  inside `mount()`. Safe across SPA navigations and dev HMR re-runs.
  Future stories that inject UI into the page reuse this pattern.
- **Disclaimer acceptance gate** (`src/lib/storage/disclaimer-store.ts`):
  reads `chrome.storage.local.disclaimer_acceptance` and matches
  `accepted:true` + `version === manifest.version`. Subscribers use
  `chrome.storage.onChanged` to react live (no need to reload the
  page when the user accepts). US-012 lands the writer.
- **Service-worker message dispatch**: stateless `onMessage` handler
  that returns `false` (synchronous `sendResponse`) — never holds the
  listener channel open across SW idle. Future SW handlers that need
  async work (e.g. captureVisibleTab in US-005) must return `true`
  AND complete via the captured `sendResponse` callback within the
  same wake.

## Hard rules for every story (enforced via globalQualityGates)

1. Manifest V3 compliance — no inline scripts, no eval, no remote code,
   service worker pattern.

2. BYOK keys never leave the user's browser to any project-controlled
   server. WebCrypto AES-GCM at rest in chrome.storage.local.

3. No site-specific selectors. No hardcoded URL allowlist beyond
   `<all_urls>`. Extraction is a pure prompt-driven Vision LLM call.
   The extension does not "target" any platform — it is a generic
   page-to-document tool.

4. No copy in the codebase, README, DISCLAIMER, PRIVACY, popup UI,
   settings page, store listing, demo video, or extracted-file metadata
   may direct the user toward any specific destination service, ATS,
   CRM, or workflow. The tool produces a file; what the user does with
   it is outside the project's scope.

5. Story-scoped commits. No unrelated files in any commit.

6. UI stories visually verified by loading dist/ as unpacked extension
   before passes=true.
