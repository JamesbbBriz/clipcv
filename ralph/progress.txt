## Codebase Patterns (will accumulate as work lands)

- **CI release workflow** (`.github/workflows/release.yml`): triggers on
  `push: tags: ['v*']`, runs on `ubuntu-latest`, `permissions: contents:
  write` so the default `GITHUB_TOKEN` can `gh release create`. Step
  sequence is `actions/checkout@v4` (with `fetch-depth: 0` so
  `package.sh`'s `git tag --list` reads the full history) →
  `actions/setup-node@v4` (`node-version: '20'`, `cache: 'npm'`) →
  `npm ci` → manifest/tag agreement check (extract `${GITHUB_REF_NAME#v}`
  and assert it equals `src/manifest.json`'s `version`) → `npm run build`
  → `npm run package` (env: `CRX_PRIVATE_KEY: ${{ secrets.CRX_PRIVATE_KEY }}`,
  `SKIP_TAG_PREFLIGHT: '1'`) → `gh release create ${GITHUB_REF_NAME}
  --draft --title 'clipcv ${GITHUB_REF_NAME}' --notes ... zip crx`.
  Releases are always **draft** so a maintainer reviews artifacts and
  writes user-facing notes before publishing.
- **`SKIP_TAG_PREFLIGHT` bypass** (`scripts/package.sh`): `package.sh`'s
  local "manifest version equals latest tag" pre-flight is a guard for
  the local "I forgot to bump" mistake. In CI, the `v*` tag IS the
  trigger and DOES equal `manifest.version` — that's the legitimate
  release state. The workflow sets `SKIP_TAG_PREFLIGHT=1` to bypass
  the check; the unset path still trips locally so US-014's invariant
  (`git tag v0.1.0 && npm run package` exits 1) holds. Future
  release-tooling stories that legitimately fire from a tagged commit
  reuse the same env-flag bypass rather than fork the script.
- **Tag-vs-manifest agreement check pattern**: extract the bare version
  from `GITHUB_REF_NAME` via shell parameter expansion
  `${GITHUB_REF_NAME#v}`, read the manifest version through an inline
  `node -e 'process.stdout.write(JSON.parse(...).version)'`, and fail
  the run if the two differ. Catches the "tagged before bumping
  manifest" mistake before any artifact is produced. The check sits
  between `npm ci` and `npm run build` so an early fail does not burn
  build time.



- **Build pipeline** is Vite 8 + `@crxjs/vite-plugin` 2.4. Manifest is the
  canonical source of build entry points: list any new background /
  content / popup / options entry in `src/manifest.json` and CRXJS picks
  it up automatically. Do NOT add `rollupOptions.input` manually.
- **Manifest path convention**: in `src/manifest.json`, refer to source
  files using their repo-relative path (`src/background/service-worker.ts`)
  for code, and the public-folder-relative path (`icons/16.png`) for
  static assets. CRXJS rewrites both to the correct `dist/...` paths at
  build time.
- **Icons live in `public/icons/`** and are emitted to `dist/icons/`.
  Placeholder 1x1 PNGs are committed for now; real 16/48/128 PNGs land
  in US-013.
- **Service worker entry**: `src/background/service-worker.ts` with
  `type: "module"` in the manifest. The SW is killed between events —
  never hold long-lived in-memory state (US-005+ message handlers must
  be event-driven).
- **TypeScript**: strict + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`.
  Path alias `@/*` → `src/*` is wired in `tsconfig.json` and `vite.config.ts`.
- **Dev server port**: 5179 (5173 collides with other local dev). HMR
  auto-rebuilds the extension; reload the extension in chrome://extensions
  to pick up SW changes.
- **No default exports** in new modules — named exports only (CLAUDE.md §9).
- **Popup entry naming**: HTML-side script ref is `./main.tsx`; the React
  component lives in `Popup.tsx`. Rationale: macOS' case-insensitive FS
  collides `popup.tsx` with `Popup.tsx`, and `forceConsistentCasingInFileNames`
  amplifies this. Future entry points (options page, content script) follow
  the same `main.tsx` + `<ComponentName>.tsx` split.
- **Tailwind**: v4 via `@tailwindcss/vite` plugin (no PostCSS config, no
  tailwind.config.js needed). Each entry point owns a `*.css` file that
  starts with `@import 'tailwindcss';`. CRXJS picks up the CSS through
  the entry HTML's `<link>` (or implicitly via the JS entry's CSS import).
- **Manifest action block**: `action.default_popup` uses the same
  `src/<entry>/<file>.html` repo-relative path; CRXJS rewrites it to
  the bundled `dist/src/<entry>/<file>.html` at build time.
- **Service worker lifecycle**: minimum coverage is
  `onInstalled` (install/update) + `onStartup` (browser launch / "activate")
  + `onSuspend` (just-before-teardown). Listeners must remain stateless —
  no module-scope mutable state.
- **Verification of UI stories**: programmatic structural check of `dist/`
  (manifest fields, popup.html shape, bundle contents, forbidden-pattern
  grep) is the documented AC verifier. Chrome 137+ blocks
  `--load-extension` from CLI without policy/dev-mode, so automated
  unpacked-load is unreliable; the build artifacts + structural checks are
  the source of truth.
- **Path alias under spaces**: the repo path `OptiTalent Nan V1/clipcv`
  contains a space. `new URL('./src', import.meta.url).pathname` returns
  the URL form (`%20`-encoded), which Rolldown can't resolve. Use
  `decodeURIComponent(...)` (or `fileURLToPath` if `@types/node` is
  available — currently it isn't) when wiring resolve.alias in
  `vite.config.ts`.
- **BYOK at-rest encryption** (`src/lib/crypto/byok-crypto.ts`): AES-GCM,
  256-bit raw key generated once per install via
  `crypto.subtle.generateKey({name:'AES-GCM', length:256}, true, ...)` and
  exported as base64; per-call random 12-byte IV. The raw key, ciphertext
  and IV all live in `chrome.storage.local` under the single key
  `byok_settings` (see `src/lib/storage/byok-store.ts`). Plaintext API
  key is never persisted. The module exports `generateRawKey`,
  `encryptString`, `decryptString` only — no shared state, no module-scope
  CryptoKey cache (the SW is killed between events).
- **WebCrypto type variance**: under TS 5.7+ `Uint8Array` defaults to
  `Uint8Array<ArrayBufferLike>`, but lib.dom's `crypto.subtle.encrypt`
  signature requires `ArrayBuffer`-backed views. We allocate a fresh
  `ArrayBuffer` copy via a small `toArrayBuffer(view)` helper at every
  WebCrypto call site — see `byok-crypto.ts`. Future stories that touch
  WebCrypto / fetch body should reuse this pattern.
- **Secret scrubber** (`src/lib/util/scrub-secrets.ts`): single regex
  pass over any string before display: `/sk-[A-Za-z0-9]{16,}/g` and
  `/Bearer\s+[A-Za-z0-9._-]{20,}/g` → `[redacted]`. Apply in every
  user-facing error path (per CLAUDE.md §6.2). The options page test-
  connection error display already routes through it; reuse the same
  helper from US-008 onwards when surfacing LLM errors.
- **Manifest permissions**: stayed minimal. Only `"storage"` requested so
  far (US-003). `<all_urls>` host permission lands with the content
  script in US-004; downloads + activeTab in US-011. Each story adds the
  narrowest required permission only.
- **Options page entry** mirrors the popup convention:
  `src/options/options.html` + `main.tsx` (createRoot bootstrap) +
  `<Component>.tsx` (form). Wired through manifest's `options_page`
  field. Tailwind imported via `options.css`. Future entry points
  (content script in US-004) follow the same `main.tsx` +
  `<Component>.tsx` split.
- **Cross-context messaging contract** (`src/lib/messaging/messages.ts`):
  every chrome.runtime.sendMessage payload is a discriminated union
  (`ClipcvMessage`) and has a matching type guard (`isClipcvMessage`).
  The service worker rejects any unknown `kind` with
  `{kind:'error', reason:'unknown_kind'}`. Receivers MUST extend the
  guard before handling new variants — do NOT add ad-hoc shape checks
  at call sites.
- **Content script structure**: entry `src/content/inject.tsx`
  (idempotent shadow-DOM mount) + component `<FloatingButton />` +
  stylesheet `inject.css`. The CSS is imported with the Vite `?inline`
  query suffix and injected as a `<style>` text node into the shadow
  root, so Tailwind utilities work inside the shadow boundary without
  leaking to the host page (no separate `inject*.css` is emitted by
  CRXJS). `:host { all: initial }` plus `host.style.all = 'initial'`
  block bidirectional CSS bleed.
- **Idempotent shadow-DOM mount**: stable host element id
  (`clipcv-floating-host`) + `getElementById` guard before insert
  inside `mount()`. Safe across SPA navigations and dev HMR re-runs.
  Future stories that inject UI into the page reuse this pattern.
- **Disclaimer acceptance gate** (`src/lib/storage/disclaimer-store.ts`):
  reads `chrome.storage.local.disclaimer_acceptance` and matches
  `accepted:true` + `version === manifest.version`. Subscribers use
  `chrome.storage.onChanged` to react live (no need to reload the
  page when the user accepts). US-012 lands the writer.
- **Service-worker message dispatch**: stateless `onMessage` handler
  that returns `false` for synchronous branches (e.g. `unknown_kind`
  rejection) and `true` for async branches (e.g. `capture-request`),
  with the async branch wrapped in a `void handle...(sender, sendResponse)`
  call so the listener returns synchronously while the handler resolves
  later. The handler MUST `sendResponse(...)` in every code path,
  including the catch block — otherwise Chrome times out the channel
  with "the message port closed before a response was received".
- **Screenshot capture pipeline** (SW-only, `src/lib/capture/screenshot.ts`):
  `chrome.tabs.captureVisibleTab(windowId?, {format:'png'})` → fetch the
  data URL → `createImageBitmap(blob)` → if width > 1280 px,
  `OffscreenCanvas(width, height*scale)` + ctx.drawImage downsample,
  then `canvas.convertToBlob({type:'image/png'})` →
  `FileReader.readAsDataURL` to base64. All four APIs are available in
  MV3 service workers; no DOM access required. `bitmap.close()` after
  use to release the GPU resource. Default cap 1280 px chosen to match
  vision-LLM tile pricing sweet spot.
- **DOM strip rule** (`src/lib/capture/dom-serialize.ts`): clone
  `document.body`, walk the subtree with `createTreeWalker(SHOW_ELEMENT)`,
  collect every `<SCRIPT>` / `<STYLE>` / `<NOSCRIPT>` AND every element
  with `aria-hidden="true"`, then detach in reverse order so parent
  removal cannot interfere with later child references. `Element.remove()`
  is a no-op when parentNode is null, so already-detached descendants
  are safe.
- **Cross-context payload shape** (`src/lib/capture/payload.ts`):
  `CapturePayload = {screenshot_b64, html, page_url, page_title, captured_at}`
  is the single contract from capture orchestrator into the downstream
  vision-extract pipeline. `applyPayloadSizeCap(payload, 1_000_000)` —
  measured via `TextEncoder().encode(JSON.stringify(payload)).byteLength` —
  drops the html field if over budget; the screenshot is preserved as
  the primary vision signal. Future stories (US-008+) consume this
  shape directly.
- **Capture orchestration topology**: content script owns the click and
  the DOM serialize; service worker owns the screenshot. Content sends
  `{kind:'capture-request'}` → SW responds with
  `{kind:'capture-screenshot', screenshot_b64, captured_at}` → content
  serializes its own DOM and assembles the final `CapturePayload`. The
  pipeline lives in `src/content/capture.ts:requestCapture()` and is
  the single entry point UI components call.
- **Manifest host_permissions**: `<all_urls>` is required for
  `chrome.tabs.captureVisibleTab` to capture any visited page. The
  `activeTab` permission is NOT a usable substitute when the trigger
  is a content-script floating button — Chrome only grants `activeTab`
  via toolbar action / context menu / keyboard shortcut gestures.
  Future stories that need broader cross-origin access already fall
  under this grant; do not redeclare.
- **LLM client topology** (`src/lib/llm/`): three-file split —
  `client.ts` (transport: timeout via AbortController, one-retry-on-5xx
  / no-retry-on-4xx, response shape extraction `choices[0].message.
  content`, typed-error mapping), `adapters.ts` (per-provider request
  shaping; `Adapter = (input) => {url, headers, body}`; current map is
  `openai/openrouter/custom` → single OpenAI-compatible builder), and
  `errors.ts` (5 typed subclasses of a private `LlmError` base:
  `AuthError`/`TimeoutError`/`RateLimitedError`/`BadRequestError`/
  `UnknownLlmError`). Adding a non-OpenAI-shaped provider = add one
  builder in adapters.ts + flip its entry in `ADAPTERS`. client.ts is
  provider-agnostic.
- **Typed-error class pattern**: each subclass sets `this.name` inside
  its own constructor — NOT via `override name = '...'` class field.
  Avoids the TS `noImplicitOverride` interaction with Error's
  interface-declared `name` (Error is declared as an interface, not a
  class, in lib.es5.d.ts) and is compatible with
  `exactOptionalPropertyTypes: true`.
- **Test runner**: vitest 4.x via `npm test` (which runs `vitest run`
  — single-shot, no watch). Tests live in
  `src/<area>/__tests__/*.test.ts`. Config in `vitest.config.ts` is
  separate from `vite.config.ts` (so vitest does not load CRXJS) and
  reuses the `decodeURIComponent(new URL('./src',
  import.meta.url).pathname)` trick to resolve `@/*` alias under the
  repo's space-containing path.
- **Injectable I/O for unit tests**: any module that does fetch /
  timeout work accepts `fetchImpl?: typeof fetch` + `timeoutMs?:
  number` opts. Defaults preserve production behaviour; tests inject
  `vi.fn` returning mocked `Response` objects. No global fetch
  monkey-patching, no msw needed yet.
- **Pre-display secret scrubber discipline**: every thrown LLM error
  message is scrubbed via `scrubSecrets()` BEFORE construction (so
  it's safe to surface in toasts / logs). Stories that introduce new
  provider-error display surfaces MUST keep this invariant.
- **Zod schema location** (`src/lib/schema/`): single source of truth
  for the LLM contract. Top-level uses `z.strictObject(...)` so the
  model cannot smuggle unknown fields past parse; nested item schemas
  (`ExperienceItemSchema`, `EducationItemSchema`) use plain
  `z.object(...)` (zod 4 default = strip unknowns) so a hallucinated
  extra field on a single experience entry does NOT poison the entire
  parse. Optional primitives are `z.string().min(1).optional()`
  (rejects empty strings), required-with-default arrays are
  `z.array(...).default([])` (model may omit; output is always a
  concrete array, no null-checks downstream).
- **Zod 4 unknown-key error shape**: strict-object rejection emits
  `{code:'unrecognized_keys', path:[], keys:[...]}` — the unknown
  field name lives in `issue.keys`, NOT `issue.path`. zod 3 used a
  path-targeted error; tests written against zod 3 idioms break here.
  When asserting "key X was rejected", look in `keys`.
- **`exactOptionalPropertyTypes` + zod**: `.optional()` produces a
  `key?: T | undefined` field. With `exactOptionalPropertyTypes: true`
  callers cannot pass `field: undefined` explicitly but can omit the
  key — fine for parse output (the parse step never emits explicit
  undefined values) and fine for upstream code that conditionally
  spreads. No special handling needed beyond avoiding `field:
  undefined` literals.
- **Vision extraction pipeline** (`src/lib/extract/vision-extract.ts`):
  single entry `extractProfile({provider, baseUrl, apiKey, model,
  screenshot_b64, html, timeoutMs?, callImpl?, fetchImpl?}) →
  ExtractedProfile`. Two-attempt loop (base prompt → strict retry on
  parse failure), `responseFormatJson:true` flag flowed end-to-end to
  the adapter, schema text inlined in the prompt for providers that
  ignore the API field, and a `{"_no_profile": true}` sentinel to
  short-circuit "this page is not a profile" → `NO_PROFILE_DETECTED`.
  `callImpl?` is the test seam — defaults to `callVisionLLM`; tests
  inject a `vi.fn` and never touch HTTP.
- **`ExtractError` shape** (`src/lib/extract/vision-extract.ts`):
  single class with a `code: ExtractErrorCode` (`AUTH | TIMEOUT |
  NO_PROFILE_DETECTED | SCHEMA_MISMATCH | UNKNOWN_LLM`) — flatter
  than the 5-subclass pattern in `llm/errors.ts` because the AC
  enumerates codes, not types, and downstream UI in US-011 will
  switch on `error.code`. Mapping from LLM transport errors:
  `AuthError → AUTH`, `TimeoutError → TIMEOUT`, everything else
  (incl. `RateLimitedError` and `BadRequestError`) → `UNKNOWN_LLM`.
- **`responseFormatJson` flag**: optional flag added to both
  `AdapterInput` and `CallVisionLLMInput`. Under
  `exactOptionalPropertyTypes:true` the propagation must be a
  conditional spread (`...(input.responseFormatJson === true ?
  { responseFormatJson: true } : {})`) — never `{ responseFormatJson:
  input.responseFormatJson }` because passing an explicit `undefined`
  is a TS error. Adapter only emits `response_format:{type:'json_object'}`
  in the body when the flag is `true`.
- **Loose JSON parser** (`parseJsonLoose` in `vision-extract.ts`):
  tries `JSON.parse(content)` first; on failure tries with surrounding
  ```` ```json ... ``` ```` code fences stripped; on failure tries the
  first `{...}` block (`indexOf('{')..lastIndexOf('}')`). Returns
  `undefined` only if all three fail. Future stories that consume LLM
  free-text output (e.g. translation, summary tasks) can reuse this
  helper if they get its own home — keep it scoped to the extract
  module for now.
- **Zod 4 issue.path PropertyKey type**: `issue.path` is
  `PropertyKey[]` (`string | number | symbol`), not just
  `(string | number)[]` like zod 3. When stringifying issue paths,
  use `path.map(String).join('.')`. Plain `path.join('.')` typechecks
  on zod 3 but fails on zod 4 because `Array<symbol>.join` is
  rejected.
- **Render layer location** (`src/lib/render/`): `pdf.ts` (US-009) and
  `docx.ts` (US-010) sit alongside a shared `filename.ts` helper.
  Filename pattern is the AC's `{Last_First_or_Unknown}_{YYYYMMDD}.{ext}` —
  multi-token names render as `Last_First` so filesystem alphabetic
  sort groups by surname; single-token names pass through; missing /
  fully-stripped names fall back to `Unknown`. Sanitization per
  CLAUDE.md §10.1 strips control chars (charcode < 0x20 or == 0x7f),
  the explicit set `/\:*?"<>|`, and `..` sequences; result capped to
  64 chars. Both renderers consume the helper via named import.
- **PDF cursor topology** (`src/lib/render/pdf.ts`): a single
  `Cursor = {pdf, page, y, font, bold}` flows through every drawing
  helper. `ensureSpace(cursor, lines, size)` checks the residual
  budget and, if insufficient, calls `pdf.addPage(...)` and resets
  `cursor.page` + `cursor.y = PAGE_HEIGHT - MARGIN`. All helpers
  decrement `cursor.y` by `size * LINE_HEIGHT` (1.4) before drawing,
  so y always points to the **top** of the next free line. Drawing
  happens at the bottom of that slot (pdf-lib uses bottom-left
  origin). `wrapText(text, font, size, maxWidth)` does word-wrap via
  `font.widthOfTextAtSize` lookups; everything bigger than max is
  pushed to a fresh line. Future renderers (DOCX paragraphs, etc.)
  do **not** need this — DOCX wraps natively — but if a future
  story adds another fixed-layout output, reuse this Cursor pattern.
- **Filename sanitization charcode trick**: literal control-char
  regex (`/[\\u0000-\\u001f]/`) in source is unsafe to write through
  the Write tool — it eats the unicode escapes and emits actual
  control bytes into the file (verified via hexdump). Use char-code
  iteration instead: `for (const ch of s) { if
  (ch.charCodeAt(0) < 0x20 || ch.charCodeAt(0) === 0x7f) continue; }`.
  Or build the regex via `new RegExp(...)` from a string literal.
  Future stories that need to filter/strip control chars must use
  one of these two approaches.
- **pdf-lib + StandardFonts**: Helvetica + HelveticaBold are emitted
  by reference (no font bytes embedded) so PDF size stays small and
  the build doesn't ship font assets. Caveat: WinAnsi (cp1252)
  encoding only — non-Latin glyphs (CJK, emoji, etc.) will throw
  inside `drawText`. Acceptable for the MVP (golden fixture is
  Latin-only); a future story that lands a custom Unicode font
  embedding (e.g. Noto Sans subset) is the long-term fix.
- **Renderer return shape**: `{ bytes: Uint8Array, filename: string }`.
  `options.now?: Date` is the determinism seam — used both in
  `setCreationDate` / `setModificationDate` (so identical input +
  identical now produces byte-identical PDFs) and in the YYYYMMDD
  filename. Tests pin `now`; the live extension passes
  `new Date()` (default). DOCX in US-010 follows the same shape.
- **DOCX renderer topology** (`src/lib/render/docx.ts`): `docx@9.x`
  + `Document` + `Paragraph` + `TextRun` + `HeadingLevel`. Builder
  helpers (`makeHeader`, `makeSectionHeading`, `makeBodyParagraph`,
  `makeExperience`, `makeEducation`, `makeListSection`, `buildBody`)
  produce flat `Paragraph[]` arrays which `buildBody` concatenates
  in the same section order as the PDF renderer. Body text is sized
  in half-points (`size: 22` = 11pt; `28` = 14pt heading; `40` = 20pt
  name). Bullet indent is twips (`360` ≈ 0.25in). Empty optional
  sections are skipped via `.length > 0` / truthy guards — same
  invariant as the PDF renderer. A trailing empty paragraph is
  appended to the section so list-only documents end with a
  well-formed terminator.
- **DOCX serialization across env**: `Packer.toBlob(doc) → Blob →
  blob.arrayBuffer() → new Uint8Array(ab)` works in BOTH the MV3
  service worker (browser Blob is built-in) AND vitest's node>=18
  env (global Blob with `.arrayBuffer()` is available since Node 18,
  no polyfill required). `Packer.toBuffer` is Node-only because it
  pipes jszip into a Buffer; do NOT use it — it breaks the browser
  bundle. The `bytes: Uint8Array` shape matches the PDF renderer.
- **DOCX byte signature**: DOCX is a ZIP container; the magic is
  `PK\\x03\\x04` (`50 4B 03 04`). Tests assert this magic plus
  `bytes.byteLength > 0`. To inspect XML inside the ZIP without
  extra deps, dynamic-import jszip (already a transitive dep of
  `docx`) — `JSZip.loadAsync(bytes)` → `zip.file('word/document.xml').
  async('string')`. Useful for asserting that section labels and
  optional-section gating actually rendered.
- **Output dispatcher** (`src/lib/render/dispatch.ts`):
  `renderForOutput(profile, output, opts)` reads the BYOK store's
  `OutputFormat` (`'pdf' | 'docx' | 'both'`) and returns
  `RenderedFile[]` with `{bytes, filename, mimeType}`. Single source
  of truth that maps the settings selector to renderer invocations;
  US-011 imports this to honour the user's choice from the floating-
  button click handler. Order is pdf-then-docx for `'both'` (matches
  the AC-defined section ordering of "PDF or DOCX or both").
  MIME types: `application/pdf` and the openxml word document type.
  Pass `{ now }` through to keep determinism in tests; default
  resolves to `new Date()`.
- **`exactOptionalPropertyTypes` + opts forwarding**: when forwarding
  a possibly-undefined `now` field into a renderer's `opts`, do NOT
  pass `{ now: options.now }` because TS rejects an explicit
  `undefined`. Use `options.now ? { now: options.now } : {}` (or
  conditional spread). Same trick used for `responseFormatJson` in
  US-008's adapter — promote this once more to a general "never
  pass `field: undefined`" rule for forwards under
  `exactOptionalPropertyTypes:true`.
- **Pipeline orchestrator location** (`src/lib/pipeline/run-pipeline.ts`):
  single entry `runPipeline({payload, signal?, onStage?, now?,
  loadByokImpl?, extractImpl?, renderImpl?, downloadImpl?, fetchImpl?})`
  → `{filenames}`. Walks `extracting → rendering → downloading → done`;
  every stage boundary calls `ensureNotAborted(signal)` so a Cancel
  click between stages collapses to `PipelineError('CANCELLED')`. If
  the LLM fetch is aborted mid-stream the orchestrator inspects
  `signal.aborted` BEFORE mapping the underlying error, so
  `AbortError` / `Error('aborted')` / etc. all surface as CANCELLED.
  ExtractError → PipelineError mapping is 1:1 on the codes
  (AUTH/TIMEOUT/NO_PROFILE_DETECTED/SCHEMA_MISMATCH/UNKNOWN_LLM);
  render and download failures wrap as RENDER_FAILED / DOWNLOAD_FAILED;
  missing settings → SETTINGS_MISSING. All thrown messages routed
  through `scrubSecrets` so sk-* / Bearer-* fragments cannot leak
  into toasts.
- **Cross-context download path**: content script wraps each
  `RenderedFile.bytes` in a `Blob({type: file.mimeType})` →
  `URL.createObjectURL(blob)` → posts
  `{kind:'download-file', url, filename}` to the SW via
  `chrome.runtime.sendMessage`. SW handles via
  `chrome.downloads.download({url, filename, saveAs: true})` and
  responds with `{kind:'download-started', downloadId}`. Blob URL is
  revoked **1 second after** sendMessage resolves — eager revoke
  aborts the download with NETWORK_FAILED because the browser process
  may not have finished reading the blob URL yet. `chrome.downloads`
  is NOT accessible from content scripts; SW is the only legal
  caller. Permissions: `"downloads"` (narrowest grant).
- **AbortController in React refs**: store the in-flight pipeline's
  `AbortController` in a `useRef`; the Cancel button calls
  `abortRef.current?.abort()`. The orchestrator reads
  `input.signal?.aborted` at every stage boundary AND, for the
  LLM-fetch step, after the catch block (so the underlying error
  type does not leak into the user toast when the user explicitly
  cancelled). Cleanup: `abortRef.current = null` in the `finally`
  block of the click handler.
- **FloatingButton state machine**: `Phase = 'idle' | 'capturing' |
  'extracting' | 'rendering' | 'downloading' | 'done'`. `ACTIVE_PHASES`
  set (capturing/extracting/rendering/downloading) drives both the
  spinner-on-button animation and the visibility of a sibling Cancel
  button. The button is `disabled` while active. `phaseLabel(phase)`
  is the only source of the user-facing label; `done` and `idle` both
  display the resting "Capture this page" copy.
- **In-shadow-DOM toast**: rendered above the button row in the same
  shadow root (no portal needed — we already own the shadow). Two
  variants (`success` emerald-50; `error` rose-50). Auto-dismissed
  after 4 s via `setTimeout(setToast(null))` whose handle is stashed
  in a `useRef` and cleared on every new toast + on unmount.
- **Pipeline error → user message** (`src/lib/pipeline/user-messages.ts`):
  single switch over `PipelineErrorCode`. Toast UI imports this
  helper exclusively; do NOT inline new copy in components. The
  test in `__tests__/run-pipeline.test.ts` pins every code → string
  pair AND greps each string for §6.4 forbidden tokens — adding a
  new code requires extending both the helper and the test in the
  same diff.
- **Message contract extension** (`src/lib/messaging/messages.ts`):
  `ClipcvMessage` is a discriminated union — to add a new kind, add
  the variant + extend `isClipcvMessage` with a per-kind branch
  that asserts the variant's required fields. Same applies to
  `ClipcvResponse`. SW dispatches each kind in its own `if
  (rawMessage.kind === ...)` branch returning `true` for async work;
  the listener's tail is an exhaustiveness fallback that returns
  `unknown_kind`.
- **Markdown-as-build-time-string** (`?raw`): for any docs file that
  must appear verbatim in the UI (e.g. `DISCLAIMER.md` rendered in
  the popup), `import md from '../../DISCLAIMER.md?raw'` returns a
  `string` typed by `vite/client.d.ts`'s wildcard `*?raw` declaration.
  Render inside `<pre className="whitespace-pre-wrap break-words
  font-sans">` so markdown characters display literally — guarantees
  "verbatim" matching to the on-disk file (mechanically verifiable
  by grepping the bundle for distinctive sentences). DO NOT add a
  markdown library — minimum blast radius wins. Imports across the
  project root (e.g. from `src/popup` to repo-root `DISCLAIMER.md`)
  work without `vite.server.fs.allow` tweaks because the project
  root is the default allow-root.
- **Disclaimer acceptance lifecycle** (`src/lib/storage/disclaimer-store.ts`):
  full read/write/clear contract — `loadDisclaimerAcceptance`,
  `isDisclaimerAccepted(version)`, `saveDisclaimerAcceptance(version,
  now=new Date())` writes `{accepted:true, version, acceptedAt:
  now.toISOString()}`, `clearDisclaimerAcceptance()` removes the key.
  On-disk shape uses camelCase `acceptedAt` (matches the existing TS
  interface). The save/clear functions land in the **shared** chunk
  (rolldown hoists shared exports), so verification scripts grep
  across popup / options / jsx-runtime bundles rather than assume
  one per-page bundle owns the call.
- **Live-reactive UI keyed off chrome.storage.local**: pattern is
  `useState<boolean | null>(null)`, `useEffect` both loads the value
  once AND subscribes to `chrome.storage.onChanged` filtered to
  `area === 'local'` AND `changes[KEY]` truthy. Any write from any
  context (the popup itself, options page, content script) flips the
  state without manual reload. Used by `FloatingButton.tsx` (US-004)
  and `Popup.tsx` (US-012). Reuse for any future store key whose UI
  needs to react cross-context.
- **Popup-takeover sizing**: the popup body has `min-width: 18rem`
  from `popup.css`; for full-takeover modes (disclaimer) set
  `w-[480px] h-[600px]` on the modal's `<main>` element directly.
  Chrome popup max is ~800x600 — 480x600 fits and is large enough
  to read multi-paragraph copy without per-line wrapping pain. No
  `popup.css` change needed.
